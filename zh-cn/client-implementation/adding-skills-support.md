本指南介绍如何为 AI 代理或开发工具添加 Agent Skills 支持。它涵盖完整的生命周期：发现技能、告知模型技能的存在、将其内容加载到上下文中，以及随着时间保持这些内容的有效性。

无论你的代理架构如何，核心集成都是一样的。实现细节因两个因素而异：

- **技能存放在哪里？** 本地运行的代理可以扫描用户的文件系统查找技能目录。云端托管或沙盒化的代理需要替代的发现机制——API、远程注册表或打包的资源。
- **模型如何访问技能内容？** 如果模型有文件读取能力，它可以直接读取 `SKILL.md` 文件。否则，你需要提供专用工具或通过程序将技能内容注入提示词。

本指南会注明这些差异产生影响的地方。你不需要支持每种场景——选择适合你代理的路径。

**前提条件**：熟悉 [Agent Skills 规范]({{ site.baseurl }}/specification.html)，它定义了 `SKILL.md` 文件格式、前置元数据字段和目录约定。

## 核心原则：渐进式披露

每个兼容技能的代理都遵循相同的三层加载策略：

| 层级 | 加载内容 | 时机 | token 成本 |
|------|---------|------|------------|
| 1. 目录 | 名称 + 描述 | 会话开始 | 每个技能约 50-100 token |
| 2. 指令 | 完整的 `SKILL.md` 正文 | 技能被激活时 | <5000 token（推荐） |
| 3. 资源 | 脚本、参考文档、资源 | 指令引用它们时 | 因情况而异 |

模型从一开始就看到目录，所以知道有哪些技能可用。当它判断某个技能相关时，会加载完整指令。如果这些指令引用了支持文件，模型会按需单独加载它们。

这保持了基础上下文的小型化，同时让模型可以按需访问专业知识。一个安装了 20 个技能的代理不会预先支付 20 个完整指令集的 token 成本——只支付实际在给定对话中使用的那些。

## 步骤 1：发现技能

在会话启动时，找到所有可用的技能并加载其元数据。

### 扫描哪些目录

扫描哪些目录取决于你的代理环境。大多数本地运行的代理至少扫描两个范围：

- **项目级别**（相对于工作目录）：特定于项目或仓库的技能。
- **用户级别**（相对于主目录）：对给定用户跨所有项目可用的技能。

其他范围也是可能的——例如，管理员部署的组织级技能，或与代理本身打包的技能。正确的范围集合取决于你的代理部署模式。

在每个范围内，考虑同时扫描**客户端特定目录**和**`.agents/skills/` 约定**：

| 范围 | 路径 | 用途 |
|------|------|------|
| 项目 | `<project>/.<your-client>/skills/` | 你的客户端的原生位置 |
| 项目 | `<project>/.agents/skills/` | 跨客户端互操作性 |
| 用户 | `~/.<your-client>/skills/` | 你的客户端的原生位置 |
| 用户 | `~/.agents/skills/` | 跨客户端互操作性 |

`.agents/skills/` 路径已成为跨客户端技能共享的广泛采用约定。虽然 Agent Skills 规范不强制规定技能目录的位置（它只定义其中的内容），但扫描 `.agents/skills/` 意味着其他兼容客户端安装的技能对你的客户端自动可见，反之亦然。

> **注意：** 一些实现也会扫描 `.claude/skills/`（项目级别和用户级别）以实现务实兼容，因为许多现有技能安装在那里。其他额外位置包括到 git root 的祖先目录（对 monorepo 有用）、[XDG](https://specifications.freedesktop.org/basedir-spec/latest/) 配置目录和用户配置的路径。

### 扫描什么

在每个技能目录中，查找**包含名为 `SKILL.md` 文件的子目录**：

```
~/.agents/skills/
├── pdf-processing/
│   ├── SKILL.md          ← 被发现
│   └── scripts/
│       └── extract.py
├── data-analysis/
│   └── SKILL.md          ← 被发现
└── README.md             ← 被忽略（不是技能目录）
```

实用的扫描规则：

- 跳过不会包含技能的目录，如 `.git/` 和 `node_modules/`
- 可选择遵循 `.gitignore` 以避免扫描构建产物
- 设置合理边界（例如最大深度 4-6 层，最多 2000 个目录）以防止在大型目录树中失控扫描

### 处理名称冲突

当两个技能共享相同的 `name` 时，应用确定性的优先规则。

现有实现的通用约定：**项目级别技能覆盖用户级别技能。**

在同一范围内（例如在 `<project>/.agents/skills/` 和 `<project>/.<your-client>/skills/` 中都发现了名为 `code-review` 的两个技能），首次发现或最后发现都是可接受的——选择一种并保持一致。当发生冲突时记录警告，让用户知道某个技能被遮蔽了。

### 信任考量

项目级别技能来自正在处理的仓库，该仓库可能不受信任（例如新克隆的开源项目）。考虑将项目级别技能加载建立在信任检查之上——只有当用户将项目文件夹标记为受信任时才加载它们。这可以防止不受信任的仓库静默地将指令注入代理的上下文。

### 云端托管和沙盒化代理

如果你的代理在容器或远程服务器中运行，它将无法访问用户的本地文件系统。发现需要根据技能范围以不同方式工作：

- **项目级别技能**通常是最简单的情况。如果代理操作克隆的仓库（即使在沙盒内），项目级别技能随代码一起传输，可以从仓库的目录树扫描。
- **用户级别和组织级别技能**在沙盒中不存在。你需要从外部来源配置它们——例如克隆配置仓库、通过代理设置接受技能 URL 或包，或让用户通过 Web UI 上传技能目录。
- **内置技能**可以作为静态资源打包在代理的部署制品中，使它们在每个会话中可用而无需外部获取。

一旦技能对代理可用，其余生命周期——解析、披露、激活——都以相同方式工作。

## 步骤 2：解析 `SKILL.md` 文件

对于每个发现的 `SKILL.md`，提取元数据和正文内容。

### 前置元数据提取

一个 `SKILL.md` 文件有两部分：`---` 分隔符之间的 YAML 前置元数据，以及关闭分隔符之后的 markdown 正文。解析方法：

1. 在文件开头找到起始 `---` 和其后的关闭 `---`。
2. 解析它们之间的 YAML 块。提取 `name` 和 `description`（必需），以及任何可选字段。
3. 关闭 `---` 之后的所有内容（修剪后）是技能的正文内容。

有关前置元数据字段及其约束的完整集合，请参见[规范]({{ site.baseurl }}/specification.html)。

### 处理格式错误的 YAML

为其他客户端编写的技能文件可能包含技术上无效但其解析器恰好接受的 YAML。最常见的问题是包含冒号的未引用值：

```yaml
# 技术上无效的 YAML —— 冒号会破坏解析
description: Use this skill when: the user asks about PDFs
```

考虑使用回退机制，在重试之前将此类值用引号包裹或转换为 YAML 块标量。这以最小代价提高了跨客户端兼容性。

### 宽松验证

对问题发出警告但尽可能仍加载技能：

- 名称与父目录名不匹配 → 警告，仍加载
- 名称超过 64 个字符 → 警告，仍加载
- 描述缺失或为空 → 跳过该技能（描述对披露至关重要），记录错误
- YAML 完全无法解析 → 跳过该技能，记录错误

记录诊断信息以便可以向用户展示（在调试命令、日志文件或 UI 中），但不要因表面问题阻止技能加载。

> **注意：** [规范]({{ site.baseurl }}/specification.html)对 `name` 字段定义了严格约束（匹配父目录、字符集、最大长度）。上面的宽松方法故意放宽这些约束以提高与其他客户端编写的技能的兼容性。

### 存储什么

至少，每个技能记录需要三个字段：

| 字段 | 描述 |
|------|------|
| `name` | 来自前置元数据 |
| `description` | 来自前置元数据 |
| `location` | `SKILL.md` 文件的绝对路径 |

将这些存储在以 `name` 为键的内存映射中，以便在激活时快速查找。

你也可以在发现时存储**正文**（前置元数据之后的 markdown 内容），或在激活时从 `location` 读取。存储使激活更快；在激活时读取在总体上使用更少内存，并且能在激活之间获取技能文件的更改。

技能的**基础目录**（`location` 的父目录）稍后需要用于解析相对路径和枚举打包资源——在需要时从 `location` 派生。

## 步骤 3：向模型披露可用技能

告诉模型有哪些技能存在，而不加载其完整内容。这是[渐进式披露的第一层](#核心原则渐进式披露)。

### 构建技能目录

对于每个发现的技能，以适合你技术栈的结构化格式包含 `name`、`description` 和可选的 `location`（`SKILL.md` 文件的路径）——XML、JSON 或项目符号列表都可以：

```xml
<available_skills>
  <skill>
    <name>pdf-processing</name>
    <description>Extract PDF text, fill forms, merge files. Use when handling PDFs.</description>
    <location>/home/user/.agents/skills/pdf-processing/SKILL.md</location>
  </skill>
  <skill>
    <name>data-analysis</name>
    <description>Analyze datasets, generate charts, and create summary reports.</description>
    <location>/home/user/project/.agents/skills/data-analysis/SKILL.md</location>
  </skill>
</available_skills>
```

`location` 字段有两个用途：它支持文件读取激活（见[步骤 4](#步骤-4激活技能)），并为模型提供了用于解析技能正文中相对引用（如 `scripts/evaluate.py`）的基础路径。如果你的专用激活工具在其结果中提供技能目录路径（见步骤 4 的[结构化包装](#结构化包装)），可以从目录中省略 `location`。否则，包含它。

每个技能向目录添加大约 50-100 token。即使安装了几十个技能，目录仍然紧凑。

### 放置目录的位置

两种常见方法：

**系统提示词部分**：在系统提示词中添加目录作为一个带标签的部分，前面加上关于如何使用技能的简要说明。这是最简单的方法，适用于任何有文件读取工具访问权限的模型。

**工具描述**：将目录嵌入专用技能激活工具的描述中（见[步骤 4](#步骤-4激活技能)）。这保持系统提示词干净，并自然地将发现与激活耦合。

两种都有效。系统提示词放置更简单、更广泛兼容；当你有专用激活工具时，工具描述嵌入更干净。

### 行为指令

在目录旁边包含简短的指令块，告诉模型如何以及何时使用技能。措辞取决于你支持哪种激活机制（见[步骤 4](#步骤-4激活技能)）：

**如果模型通过读取文件激活技能：**

```
The following skills provide specialized instructions for specific tasks.
When a task matches a skill's description, use your file-read tool to load
the SKILL.md at the listed location before proceeding.
When a skill references relative paths, resolve them against the skill's
directory (the parent of SKILL.md) and use absolute paths in tool calls.
```

**如果模型通过专用工具激活技能：**

```
The following skills provide specialized instructions for specific tasks.
When a task matches a skill's description, call the activate_skill tool
with the skill's name to load its full instructions.
```

保持这些指令简洁。目标是告诉模型技能存在以及如何加载它们——技能内容本身在加载后提供详细指令。

### 过滤

某些技能应从目录中排除。常见原因：

- 用户在设置中禁用了该技能
- 权限系统拒绝访问该技能
- 技能选择退出模型驱动的激活（例如通过 `disable-model-invocation` 标志）

**完全隐藏被过滤的技能**，而不是列出它们并在激活时阻止。这防止模型浪费轮次尝试加载它无法使用的技能。

### 当没有可用技能时

如果没有发现技能，完全省略目录和行为指令。不要显示空的 `<available_skills/>` 块或注册没有有效选项的技能工具——这会混淆模型。

## 步骤 4：激活技能

当模型或用户选择技能时，将完整指令传递到对话上下文中。这是[渐进式披露的第二层](#核心原则渐进式披露)。

### 模型驱动的激活

大多数实现依赖模型自身的判断作为激活机制，而不是实现框架端的触发器匹配或关键词检测。模型读取目录（来自[步骤 3](#步骤-3向模型披露可用技能)），判断某个技能与当前任务相关，并加载它。

两种实现模式：

**文件读取激活**：模型使用目录中的 `SKILL.md` 路径调用其标准文件读取工具。不需要特殊基础设施——代理现有的文件读取能力就足够了。模型接收文件内容作为工具结果。当模型有文件访问权限时，这是最简单的方法。

**专用工具激活**：注册一个工具（例如 `activate_skill`），接受技能名称并返回内容。当模型无法直接读取文件时这是必需的，即使可以也是有用的（但可选）。相比原始文件读取的优势：

- 控制返回什么内容——例如去除 YAML 前置元数据或保留它（见下面的[模型接收什么](#模型接收什么)）
- 用结构化标签包装内容以便在上下文管理期间识别
- 在指令旁边列出打包资源（例如 `references/*`）
- 强制权限或提示用户同意
- 跟踪激活用于分析

> **提示：** 如果你使用专用激活工具，将 `name` 参数约束为有效技能名称的集合（例如作为工具模式中的枚举）。这防止模型幻觉出不存在的技能名称。如果没有可用技能，根本不要注册该工具。

### 用户显式激活

用户也应该能够直接激活技能，而无需等待模型决定。最常见的模式是**斜杠命令或提及语法**（`/skill-name` 或 `$skill-name`），由框架拦截。具体语法由你决定——关键思想是框架处理查找和注入，所以模型接收技能内容而不需要自己执行激活操作。

自动补全小部件（在用户输入时列出可用技能）也可以使这变得可发现。

### 模型接收什么

当技能被激活时，模型接收技能的指令。该内容具体是什么样的有两个选项：

**完整文件**：模型看到整个 `SKILL.md`，包括 YAML 前置元数据。这是文件读取激活的自然结果，模型读取原始文件。这也是专用工具的有效选择。前置元数据可能包含在激活时有用的字段——例如 [`compatibility`]({{ site.baseurl }}/specification.html#compatibility-field) 注明了环境要求，可以影响模型执行技能指令的方式。

**仅正文（去除前置元数据）**：框架解析并移除 YAML 前置元数据，只返回 markdown 指令。在有专用激活工具的现有实现中，大多数采用这种方法——在发现期间提取 `name` 和 `description` 后去除前置元数据。

两种方法在实践中都有效。

### 结构化包装

如果你使用专用激活工具，考虑用识别标签包装技能内容。例如：

```xml
<skill_content name="pdf-processing">
# PDF Processing

## When to use this skill
Use this skill when the user needs to work with PDF files...

[SKILL.md 正文的其余部分]

Skill directory: /home/user/.agents/skills/pdf-processing
Relative paths in this skill are relative to the skill directory.

<skill_resources>
  <file>scripts/extract.py</file>
  <file>scripts/merge.py</file>
  <file>references/pdf-spec-summary.md</file>
</skill_resources>
</skill_content>
```

这有实际好处：

- 模型可以清楚地区分技能指令和其他对话内容
- 框架可以在上下文压缩期间识别技能内容（[步骤 5](#步骤-5随时间管理技能上下文)）
- 打包资源被呈现给模型而不被急切加载

### 列出打包资源

当专用激活工具返回技能内容时，它也可以枚举技能目录中的支持文件（脚本、参考文档、资源）——但**不应急切读取它们**。模型在技能指令引用时使用其文件读取工具按需加载特定文件。

对于大型技能目录，考虑限制列表并注明可能不完整。

### 权限白名单

如果你的代理有控制文件访问的权限系统，**将技能目录加入白名单**，这样模型可以读取打包资源而不触发用户确认提示。否则，每次引用打包脚本或参考文件都会导致权限对话框，破坏包含 `SKILL.md` 以外资源的技能的流程。

## 步骤 5：随时间管理技能上下文

一旦技能指令进入对话上下文，在会话期间保持其有效性。

### 保护技能内容免受上下文压缩

如果你的代理在上下文窗口填满时截断或总结旧消息，**将技能内容豁免于修剪**。技能指令是持久的行为指导——在对话中途丢失它们会静默降低代理性能而没有任何可见错误。模型继续运行但没有技能提供的专业指令。

常见方法：

- 将技能工具输出标记为受保护，以便修剪算法跳过它们
- 使用步骤 4 中的[结构化标签](#结构化包装)识别技能内容并在压缩期间保留它

### 去重激活

考虑跟踪当前会话中哪些技能已被激活。如果模型（或用户）尝试加载已在上下文中的技能，你可以跳过重新注入以避免相同指令在对话中出现多次。

### 子代理委托（可选）

这是一种高级模式，只有部分客户端支持。技能不在主对话中注入指令，而是在**单独的子代理会话**中运行。子代理接收技能指令、执行任务并向主对话返回其工作摘要。

当技能的工作流足够复杂以至于受益于专用、专注的会话时，这种模式很有用。
