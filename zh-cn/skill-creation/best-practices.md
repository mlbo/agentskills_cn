## 从真实专业知识出发

创建技能时的一个常见错误是让 LLM 在没有提供领域特定上下文的情况下生成技能——仅依赖 LLM 的通用训练知识。结果是模糊、通用的流程（如"适当处理错误"、"遵循认证最佳实践"），而不是让技能具有价值的特定 API 模式、边界情况和项目约定。

有效的技能需要建立在真实专业知识的基础上。关键是将领域特定上下文输入到创建过程中。

### 从实际任务中提取

在与代理的对话中完成一个真实任务，沿途提供上下文、纠正和偏好。然后将可复用的模式提取为技能。注意：

- **有效的步骤** —— 导致成功的操作序列
- **你做的纠正** —— 你引导代理方法的地方（例如，"使用库 X 而不是 Y"、"检查边界情况 Z"）
- **输入/输出格式** —— 数据输入和输出时的样子
- **你提供的上下文** —— 代理不知道的项目特定事实、约定或约束

### 从现有项目产物中综合

当你有一定量的现有知识时，可以将其输入 LLM 并要求它综合出一个技能。从团队实际的事故报告和运维手册综合出的数据管道技能，会比从通用的"数据工程最佳实践"文章中综合出的技能更好，因为它捕获了*你的*模式、失败模式和恢复流程。关键是项目特定的材料，而不是通用的参考。

好的源材料包括：

- 内部文档、运维手册和风格指南
- API 规范、模式和配置文件
- 代码审查评论和问题跟踪器（捕获重复出现的问题和审查者的期望）
- 版本控制历史，特别是补丁和修复（通过实际更改的内容揭示模式）
- 真实世界的失败案例及其解决方案

## 通过实际执行进行优化

技能的初稿通常需要优化。针对真实任务运行技能，然后将结果——所有结果，不仅仅是失败——反馈到创建过程中。问自己：什么触发了误报？遗漏了什么？可以删减什么？

即使是一轮执行后修订也能明显提高质量，复杂的领域通常需要多轮迭代。

> **提示：** 阅读代理执行轨迹，而不仅仅是最终输出。如果代理在无效步骤上浪费时间，常见原因包括指令太模糊（代理在找到有效方法之前尝试了多种方法）、指令不适用于当前任务（代理仍然遵循它们），或者提供了太多选项而没有明确的默认值。

对于更结构化的迭代方法，包括测试用例、断言和评分，请参阅[评估技能输出质量]({{ site.baseurl }}/skill-creation/evaluating-skills.html)。

## 明智地使用上下文

当技能激活时，其完整的 `SKILL.md` 正文会加载到代理的上下文窗口中，与对话历史、系统上下文和其他激活的技能一起。技能中的每个 token 都在与该窗口中的其他所有内容竞争代理的注意力。

### 添加代理缺少的内容，省略代理已知的内容

专注于代理*没有*你的技能就不会知道的内容：项目特定约定、领域特定流程、非明显的边界情况，以及要使用的特定工具或 API。你不需要解释什么是 PDF、HTTP 如何工作，或数据库迁移做什么。

````markdown
<!-- 太冗长 —— 代理已经知道什么是 PDF -->
## 提取 PDF 文本

PDF（便携式文档格式）文件是一种常见的文件格式，包含
文本、图像和其他内容。要从 PDF 中提取文本，你需要
使用一个库。推荐使用 pdfplumber，因为它能很好地处理大多数情况。

<!-- 更好 —— 直接跳到代理自己不知道的内容 -->
## 提取 PDF 文本

使用 pdfplumber 进行文本提取。对于扫描文档，回退到
pdf2image 和 pytesseract。

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

对每段内容问自己："没有这个指令，代理会做错吗？"如果答案是否定的，删掉它。如果不确定，测试一下。如果代理在没有技能的情况下已经能很好地处理整个任务，那么这个技能可能没有增加价值。请参阅[评估技能输出质量]({{ site.baseurl }}/skill-creation/evaluating-skills.html)了解如何系统地测试这一点。

### 设计连贯单元

决定技能应该覆盖什么，就像决定函数应该做什么一样：你希望它封装一个能与其他技能良好组合的连贯工作单元。范围太窄的技能会强制为单个任务加载多个技能，带来开销和指令冲突的风险。范围太宽的技能难以精确激活。一个用于查询数据库并格式化结果的技能可能是一个连贯单元，而一个还覆盖数据库管理的技能可能做得太多了。

### 追求适度细节

过于全面的技能可能弊大于利——代理难以提取相关内容，可能会被不适用于当前任务的指令误导而走上有问题的路径。简洁、分步骤的指导加上一个可行的示例，往往优于详尽的文档。当你发现自己在覆盖每个边界情况时，考虑大多数是否由代理自己的判断来处理更好。

### 用渐进式披露构建大型技能

[规范]({{ site.baseurl }}/specification.html#progressive-disclosure)建议将 `SKILL.md` 保持在 500 行和 5,000 个 token 以下——只包含代理每次运行所需的核心指令。当技能确实需要更多内容时，将详细的参考材料移到 `references/` 或类似目录中的单独文件中。

关键是告诉代理*何时*加载每个文件。"如果 API 返回非 200 状态码，读取 `references/api-errors.md`"比通用的"详见 references/"更有用。这让代理按需加载上下文，而不是预先加载，这正是[渐进式披露]({{ site.baseurl }}/what-are-skills.html#how-skills-work)的设计方式。

## 校准控制

技能的每个部分不需要同样级别的详细程度。根据任务的脆弱性匹配指令的特异性。

### 匹配特异性到脆弱性

**给代理自由度**，当多种方法都有效且任务容许变化时。对于灵活的指令，解释*为什么*比僵化的命令更有效——理解指令目的的代理能做出更好的上下文相关决策。代码审查技能可以描述要寻找什么，而不需要规定确切的步骤：

```markdown
## 代码审查流程

1. 检查所有数据库查询是否有 SQL 注入（使用参数化查询）
2. 验证每个端点上的认证检查
3. 查找并发代码路径中的竞态条件
4. 确认错误消息不会泄露内部细节
```

**要明确**，当操作脆弱、一致性重要、或必须遵循特定序列时：

````markdown
## 数据库迁移

精确运行此序列：

```bash
python scripts/migrate.py --verify --backup
```

不要修改命令或添加额外的标志。
````

大多数技能是混合的。独立校准每个部分。

### 提供默认值，而不是菜单

当多种工具或方法都可行时，选择一个默认值并简要提及其他选项，而不是将它们作为同等选项呈现。

````markdown
<!-- 选项太多 -->
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image...

<!-- 明确的默认值加备选方案 -->
使用 pdfplumber 进行文本提取：

```python
import pdfplumber
```

对于需要 OCR 的扫描 PDF，改用 pdf2image 和 pytesseract。
````

### 偏好流程而非声明

技能应该教代理*如何处理*一类问题，而不是针对特定实例*生产什么*。比较：

```markdown
<!-- 特定答案 —— 只对这个确切任务有用 -->
将 `orders` 表与 `customers` 表在 `customer_id` 上连接，筛选
`region = 'EMEA'`，并对 `amount` 列求和。

<!-- 可复用方法 —— 适用于任何分析查询 -->
1. 从 `references/schema.yaml` 读取模式以找到相关表
2. 使用 `_id` 外键约定连接表
3. 将用户请求中的任何筛选器应用为 WHERE 子句
4. 按需聚合数值列并格式化为 markdown 表格
```

这并不意味着技能不能包含特定细节——输出格式模板（见[输出格式模板](#输出格式模板)）、像"永不输出 PII"这样的约束，以及工具特定指令都很有价值。关键在于*方法*应该泛化，即使个别细节是特定的。

## 有效指令的模式

这些是构建技能内容的可复用技术。不是每个技能都需要所有这些——使用适合你任务的部分。

### 陷阱部分

许多技能中最高价值的内容是陷阱列表——违反合理假设的环境特定事实。这些不是通用建议（如"适当处理错误"），而是对代理在没有被告知的情况下会犯的错误的具体纠正：

````markdown
## 陷阱

- `users` 表使用软删除。查询必须包含
  `WHERE deleted_at IS NULL`，否则结果将包含已停用的账户。
- 用户 ID 在数据库中是 `user_id`，在认证服务中是 `uid`，
  在计费 API 中是 `accountId`。这三者指向同一个值。
- `/health` 端点只要 Web 服务器在运行就返回 200，
  即使数据库连接已断开。使用 `/ready` 检查完整
  服务健康状态。
````

将陷阱保留在 `SKILL.md` 中，让代理在遇到情况之前就读取它们。单独的参考文件可行，前提是你告诉代理何时加载它，但对于非明显的问题，代理可能识别不出触发条件。

> **提示：** 当代理犯了你需要纠正的错误时，将纠正添加到陷阱部分。这是迭代改进技能最直接的方式之一（见[通过实际执行进行优化](#通过实际执行进行优化)）。

### 输出格式模板

当你需要代理以特定格式生成输出时，提供模板。这比用散文描述格式更可靠，因为代理擅长对具体结构进行模式匹配。短模板可以内联在 `SKILL.md` 中；对于较长的模板或仅在某些情况下需要的模板，存储在 `assets/` 中并从 `SKILL.md` 引用，这样只在需要时加载。

````markdown
## 报告结构

使用此模板，根据具体分析调整各部分：

```markdown
# [分析标题]

## 执行摘要
[关键发现的一段概述]

## 主要发现
- 发现 1 及支持数据
- 发现 2 及支持数据

## 建议
1. 具体可操作的建议
2. 具体可操作的建议
```
````

### 多步骤工作流的清单

明确的清单帮助代理跟踪进度和避免跳过步骤，特别是当步骤有依赖关系或验证关口时。

```markdown
## 表单处理工作流

进度：
- [ ] 步骤 1：分析表单（运行 `scripts/analyze_form.py`）
- [ ] 步骤 2：创建字段映射（编辑 `fields.json`）
- [ ] 步骤 3：验证映射（运行 `scripts/validate_fields.py`）
- [ ] 步骤 4：填写表单（运行 `scripts/fill_form.py`）
- [ ] 步骤 5：验证输出（运行 `scripts/verify_output.py`）
```

### 验证循环

指导代理在继续之前验证自己的工作。模式是：做工作，运行验证器（脚本、参考清单或自查），修复任何问题，重复直到验证通过。

```markdown
## 编辑工作流

1. 进行编辑
2. 运行验证：`python scripts/validate.py output/`
3. 如果验证失败：
   - 查看错误消息
   - 修复问题
   - 再次运行验证
4. 只有在验证通过后才继续
```

参考文档也可以作为"验证器"——指导代理在最终确定前根据参考检查其工作。

### 计划-验证-执行

对于批量或破坏性操作，让代理以结构化格式创建中间计划，根据事实来源验证它，然后才执行。

```markdown
## PDF 表单填写

1. 提取表单字段：`python scripts/analyze_form.py input.pdf` → `form_fields.json`
   （列出每个字段名称、类型以及是否必填）
2. 创建 `field_values.json`，将每个字段名称映射到其预期值
3. 验证：`python scripts/validate_fields.py form_fields.json field_values.json`
   （检查每个字段名称是否存在于表单中、类型是否兼容、
   必填字段是否缺失）
4. 如果验证失败，修改 `field_values.json` 并重新验证
5. 填写表单：`python scripts/fill_form.py input.pdf field_values.json output.pdf`
```

关键要素是第 3 步：一个验证脚本，检查计划（`field_values.json`）与事实来源（`form_fields.json`）。像"字段 'signature_date' 未找到 —— 可用字段：customer_name、order_total、signature_date_signed"这样的错误给代理足够的信息来自我纠正。

### 打包可复用脚本

当[迭代技能]({{ site.baseurl }}/skill-creation/evaluating-skills.html)时，比较代理在测试用例间的执行轨迹。如果你注意到代理每次运行都在独立重新发明相同的逻辑——构建图表、解析特定格式、验证输出——那就是信号，应该编写一次测试过的脚本并打包在 `scripts/` 中。

更多关于设计和打包脚本的内容，请参阅[在技能中使用脚本]({{ site.baseurl }}/skill-creation/using-scripts.html)。
